## 0. 이 문서 사용법 (개발자/Claude Code용)

이 문서는 청첩장용 웹 미니게임의 개발 요구사항을 정리한 PRD입니다. **MVP(Step 1~2)만 우선 완성**하고, 이후 단계는 여유가 있을 때 추가합니다. 결혼식 날짜라는 고정 마감일이 있으므로 **완성도 > 기능 개수**입니다.

## 1. 프로젝트 개요

- **제품명**: 웨딩러너 (가칭)
- **유형**: 모바일 웹 기반 미니게임 (청첩장 QR로 진입)
- **컨셉**: 포켓몬 1세대 도트 감성 오마주 + 웨딩 테마 런게임
- **목적**:
  - 하객 참여형 이벤트 (TOP 3 경품 증정)
  - 결혼식장 대형 스크린 실시간 리더보드 송출
  - 단톡방 바이럴로 하객 간 점수 경쟁 유도

## 2. 성공 지표

- 플레이당 소요 시간: 1~2분 (스토리 스킵 시 60초)
- 목표 참여율: 하객의 30% 이상
- 재플레이율: 1인당 평균 3회 이상
- 카톡 공유 발생률: 플레이어의 20% 이상

## 3. 기술 스택

| 영역 | 기술 | 이유 |
|---|---|---|
| 게임 엔진 | **Phaser 3 + TypeScript** | 2D 도트 게임에 최적, 모바일 웹 대응 우수 |
| 번들러 | **Vite** (PRD 원안) / 본 프로젝트는 **Next.js 16** 유지 | 기존 청첩장이 Next.js. Phaser는 dynamic import + ssr:false |
| 백엔드 | **Firebase Firestore** | 서버리스 실시간 리더보드 |
| 공유 기능 | **html2canvas + Kakao JavaScript SDK** | 결과 카드 이미지화 + 카톡 공유 |
| 배포 | **Vercel** | 정적 웹 호스팅, HTTPS 자동, 기존 알리아스 유지 |

## 5. MVP 기능 명세 (필수 구현)

### 5.1 웨딩러너 본게임 (Phase 4)

**시점 및 조작**
- 탑다운 2D 시점, 세로 스크롤
- 레인 수: **3레인** (좌/중/우)
- 모바일 조작: 화면 **좌측 절반 탭 = 왼쪽 이동**, **우측 절반 탭 = 오른쪽 이동**
- 좌우 스와이프도 인식 (보조 입력)
- 데스크탑: 좌우 방향키 / A·D 키
- 캐릭터는 자동 전진 (실제 구현: 배경이 아래로 스크롤)

**시간 구조 (60초)**

| 구간 | 시간 | 설명 |
|---|---|---|
| 튜토리얼 | 0~15초 | 장애물 밀도 낮음, 아이템 위주, 규칙 학습 |
| 본 게임 | 15~40초 | 정상 난이도, 아이템과 장애물 균형 |
| 클라이맥스 | 40~60초 | 밀도 상승, 하객 박수 이펙트 강화, BGM 고조 |
| 엔딩 | 60초 | 자동으로 제단 도착, 진화 연출 |

### 5.2 아이템 시스템

| 아이템 | 점수 | 등장 빈도 | 효과 |
|---|---|---|---|
| 하트 ❤️ | +30 | 매우 자주 | - |
| 반지 💍 | +100 | 자주 | - |
| 부케 💐 | +250 | 드물게 | - |
| 샴페인 🥂 | +80 | 드물게 | **3초간 무적** (캐릭터 반짝임) |

### 5.3 장애물 시스템

| 장애물 | 효과 |
|---|---|
| 깨진 샴페인잔 | 감점 -80, 캐릭터 잠시 멈칫 |
| 뛰어다니는 화동 | 감점 -150, **2초간 좌우 조작 반전 (혼란 상태)** |
| 넘어진 웨딩케이크 | **게임오버** (단, 샴페인 무적 중이면 무시) |

**중요**: 게임오버 유발 장애물은 케이크 1종으로 고정. 운빨 요소 최소화.

### 5.4 점수 공식

```
최종 점수 = 아이템 점수 합
          + (생존 초 × 10)
          + 콤보 보너스
          - 감점 합
```

**콤보 보너스** (연속으로 아이템만 획득 시)
- 5연속: +50
- 10연속: +150
- 장애물에 부딪히면 콤보 초기화

### 5.5 닉네임 & 리더보드

- 최초 진입 시 닉네임 입력 (2~8자 한글/영문/숫자)
- `localStorage`에 닉네임 + 고유 ID(UUID) 저장 → 재진입 시 유지
- Firestore에는 **최고 점수만 갱신** (Insert가 아닌 Upsert)
- 닉네임 중복 허용 (고유 ID로 구분)
- 리더보드 화면:
  - 상위 10위 표시
  - 본인이 TOP 10에 없으면 하단에 본인 순위 별도 표시
  - 본인 행은 하이라이트

### 5.6 결과 카드 + 카톡 공유 ⭐ 바이럴 핵심

**결과 카드 디자인 (도감 스타일)**

```
┌─────────────────────┐
│  No.042  김하객      │
│  ─────────────      │
│  [도트 캐릭터 일러]  │
│                     │
│  점수: 2,450        │
│  최대 콤보: 7       │
│  순위: 23위         │
│  "효과는 굉장했다!"  │
└─────────────────────┘
```

**구현 순서**
1. 결과 카드를 HTML/CSS로 렌더링
2. `html2canvas`로 캔버스 캡처 → PNG 이미지 생성
3. 카카오톡 Feed 공유 API로 단톡방 전송
4. 카톡 공유 실패 시 폴백: "이미지 저장하기" 버튼

**필수 버튼**
- [카카오톡으로 자랑하기]
- [이미지 저장]
- [리더보드 보기]
- [다시 하기]

## 6. 데이터 구조

### 6.1 Firestore 스키마

```typescript
// Collection: scores
{
  playerId: string       // localStorage의 UUID
  nickname: string       // 표시용 닉네임 (최대 8자)
  score: number          // 최종 점수
  itemScore: number      // 아이템 획득 점수 (검증용)
  maxCombo: number       // 최대 콤보 (검증용)
  survivalTime: number   // 생존 시간 초 단위 (검증용)
  timestamp: Timestamp   // 서버 시간
  gameVersion: string    // 게임 버전 (예: "1.0.0")
}
```

**쿼리**
- 리더보드: `orderBy("score", "desc").limit(10)`
- 본인 순위: `where("playerId", "==", myId)`

### 6.2 localStorage 키

```
wr_player_id: string       // UUID (최초 진입 시 생성)
wr_nickname: string        // 입력한 닉네임
wr_best_score: number      // 로컬 최고 점수
wr_seen_tutorial: boolean  // 튜토리얼 본 적 있는지
```

## 8. 모바일 최적화

- **기준 해상도**: 480 × 854 (세로 9:16)
- Phaser Scale 설정: `Scale.FIT` + `Scale.CENTER_BOTH`
- 터치 영역: 화면 좌/우 절반 각각 (시각적 인디케이터 표시)
- 최소 폰트 크기: 14px
- iOS Safari 오디오 제약 대응: **첫 사용자 터치 후 오디오 재생 시작**
- 한손 조작 고려: 중요 버튼은 화면 하단 배치
- Phase 1~3에는 **우측 상단 [SKIP ⏭] 버튼 상시 노출**

## 9. 치팅 완화 (완벽 방지가 아닌 이상치 필터링)

- 점수 제출 시 **플레이 메타데이터 함께 전송** (생존시간, 아이템 수, 콤보 수)
- **서버측 검증**:
  - 60초 미만 생존에서 최대 가능 점수 계산 → 초과 시 거부
  - 동일 playerId로 10초 이내 반복 제출 차단
  - 이상치 점수는 리더보드에서 자동 제외
